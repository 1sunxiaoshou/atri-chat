# 未来改造参考

本文档只保留旧方案中仍可能有价值的想法。这里的内容不是当前事实，也不是承诺计划；进入实施前必须重新对照当前代码和 [路线图](./路线图.md) 决策。

## 1. VRM 分层思路

旧方案中“对话层、指挥层、渲染层”的拆分仍可作为未来重构参考：

- 对话层负责接收 Agent 输出和 custom event。
- 指挥层负责把意图编排为前端可执行动作。
- 渲染层只关心 R3F、VRM、动作、表情和镜头状态。

当前实现还没有独立完整的 Director，因此不能把旧方案中的类名、目录结构和状态机视为事实。

## 2. 表情与音频分层

可保留的思路：

- 情感表情、口型、眨眼/注视等自动行为应尽量解耦。
- TTS 语音播放和字幕展示需要与 `say` 命令保持顺序关系。
- 纯渲染层自治逻辑不一定需要经过 Agent 或后端。

当前实现仍以 `useVRM` 为主要执行入口，后续若拆分，应先保证命令系统行为不变。

## 3. 场景配置

未来可以继续评估：

- 背景配置
- 相机预设
- 角色默认站位
- 渲染效果预设

这类能力应优先从现有 `VRMRenderSettings` 和 `useVRMStore` 渐进扩展，不建议照搬旧方案的大目录重写。

## 4. 桌宠模式

桌宠模式仍是可能的产品方向，但不进入近期主线。

未来若推进，应先确认：

- Tauri 多窗口是否稳定满足透明、置顶、穿透等需求。
- 主窗口与桌宠窗口如何同步角色状态。
- 是否复用轻量 VRM 预览组件。
- 桌宠模式是否需要独立交互、独立会话或只作为镜像展示。

## 5. 能力生命周期

旧方案中“能力注册、状态展示、按需初始化”的方向仍有价值。当前已有最小 `CapabilityRegistry`，后续应基于现有实现渐进增强：

- 统一启动、预热、失败、重置状态。
- 明确 ASR、TTS、Agent、VRM 的生命周期边界。
- 谨慎决定是否持久化运行事件。

## 6. 使用原则

使用本文时遵守以下规则：

- 不把旧类名、旧目录结构当作当前架构。
- 不恢复已废弃通信链路。
- 不恢复文本内嵌标签式命令协议。
- 不以一次性大重写替代渐进整理。
- 任何内容进入实施前，先更新 [路线图](./路线图.md)。
