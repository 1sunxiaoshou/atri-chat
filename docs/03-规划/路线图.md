# 路线图

本文档是 ATRI Chat 唯一活跃计划来源，只记录已经决定推进、暂缓或停止的事项。当前实现以 [系统设计](../02-架构/系统设计.md) 和各模块架构文档为准；候选思路见 [未来改造参考](./未来改造参考.md)，历史风险分析见 [架构审计与重构建议](./架构审计与重构建议.md)。

## 1. 当前目标

当前发布目标是建立可信、可恢复、可验证的 Windows 桌面基线。优先保护的主链是：

```text
Tauri 启动
 -> FastAPI /health
 -> /agent-stream
 -> useAgentStream
 -> 文本聊天 / VRM 命令执行
 -> runtime status / feedback
```

实施顺序：

```text
安全与数据正确性
 -> Agent run 正确性
 -> VRM 执行可靠性
 -> 产品体验收口
 -> 新能力评审
```

路线图条目只有满足验收标准后才算完成。完成后的当前事实写入 `docs/02-架构/`，不在本文长期保留完成清单。

## 2. M0：发布安全基线

### 2.1 桌面后端可信边界

目标：把 Python sidecar 从局域网服务收口为只接受当前桌面会话访问的本地服务。

- 后端只监听 loopback。
- Tauri 每次启动生成会话 token，并注入 Python 与前端请求层。
- CORS 和 CSP 使用实际需要的最小允许范围。
- Provider、TTS 配置接口不返回完整密钥。
- 上传接口统一限制大小、扩展名和内容类型。

验收：非本机和无有效 token 的请求不能访问业务 API；列表和详情接口不会泄露完整密钥；异常上传不会无限占用内存或磁盘。

### 2.2 会话生命周期一致性

目标：用户看到的会话、业务消息和 LangGraph checkpoint 具有一致的清空与删除语义。

- 定义清空消息、删除会话、长期记忆保留与标题保留规则。
- 由统一用例清理业务消息和对应 checkpoint。
- 为清空、删除和重复清理补集成测试。

验收：清空后的下一轮上下文不再包含旧消息；删除后不产生 checkpoint 孤儿；失败的清理可重试。

### 2.3 发布质量门

目标：一次发布预检覆盖当前已有质量检查。

- 串联 Ruff、Black、Pytest、TypeScript、ESLint、i18n、命令解析、版本、锁文件和资源检查。
- 任一检查失败时返回非零退出码并阻止发布。

验收：本地和 CI 使用同一预检入口，失败项可直接定位。

## 3. M1：Agent run 正确性

### 3.1 Run 与 turn 生命周期

- 增加可查询的 run/turn 状态，区分 running、completed、failed 和 cancelled。
- 用户、AI 与工具消息按 turn 批量持久化，并保留幂等键。
- 支持停止生成、失败重试和中断恢复，避免重复消息。

验收：任一消息可追溯到 turn；stream 中断后状态明确；重试不会重复持久化。

### 3.2 实时事件协议

- 事件包含协议版本、conversation ID、run ID、turn ID、sequence 和明确事件类型。
- ActionCommand 改为结构化数据；旧字符串解析器只作为短期兼容层。
- Python schema 与 TypeScript DTO 建立单一生成或校验来源。

验收：前端能够去重、排序并关联执行反馈；新增事件类型可以被穷尽检查。

### 3.3 会话查询

- 会话列表使用聚合计数，不加载每个会话的全部消息。
- 消息接口使用稳定 cursor，首次加载最新一页并支持向前翻页。

验收：查询数据量随会话数和页大小增长，不随全部历史消息总量增长。

## 4. M2：VRM 执行可靠性

### 4.1 VRM Director

- 建立单消费者队列，统一处理 action batch 和相机命令。
- 明确 motion、say、wait、camera 的完成、超时、取消和抢占语义。
- 新 run 可以取消旧动作与音频，结束后可靠回到 idle。
- 反馈按 run、tool call 和 command 关联。

验收：连续批次不会并发争抢音频或运行状态；每条命令都有 completed、failed 或 cancelled 结果。

### 4.2 TTS 执行与缓存

- TTS 不再阻塞 action event 首次发送。
- Provider 实例缓存与音频内容缓存分离。
- 音频缓存键包含供应商、配置版本、音色、语言和规范化文本，并提供淘汰策略。

验收：切换角色或音色不会串音；TTS 失败不阻止非语音动作；首次动作不等待全部语音合成。

### 4.3 VRM 资源与性能

- 明确 GLTF/VRM 缓存与 dispose 的唯一所有者。
- 关闭常驻 `preserveDrawingBuffer`，恢复合理的视锥剔除。
- 增加 DPR、FPS 和窗口不可见时的降级策略。

验收：重复切换模型不会使用已释放缓存；低性能设备可降级；VRM 帧时间可观察。

## 5. M3：产品体验收口

### 5.1 零配置与设置体验

- 明确首次启动、无角色、无模型、无音色和无动作时的界面与引导。
- 整理默认头像、背景、动作和音色的兜底边界。
- 优化设置页的信息分组和错误反馈。

验收：新数据目录启动后，用户能够明确知道下一步，并能在不进入失效页面的情况下完成首次对话配置。

### 5.2 前端状态边界

- 从 `App.tsx` 提取会话选择与加载职责。
- 区分服务端数据、持久化设置、会话 UI、Agent run 和 VRM 高频渲染状态。
- 把仍在使用聚合 `api` 或直接 `httpClient` 的调用迁移到模块化 API。
- 继续使用现有 React、Zustand 和 R3F，不把更换状态库作为目标。

验收：会话、角色和模型切换只有一个状态拥有者；关键流程不依赖跨组件手工同步；业务组件不再调用兼容聚合 API。

### 5.3 VRM 产品功能

- 支持头像位置显示切换。
- 支持为角色选择允许 AI 使用的表情子集。
- 在性能预算内继续调整卡通渲染观感。
- 把现有背景与相机控制收口为稳定设置和预设。

验收：配置有持久化入口、即时预览和明确兜底；Prompt 只暴露启用的表情。

## 6. 候选方向

以下方向尚未承诺实施。进入主线前必须先形成产品目标、技术边界和验收标准，再更新本路线图：

- 图片、音频、屏幕捕获等多模态输入。
- 桌宠或透明挂件模式。
- Live2D。
- OBS 输出。
- 角色、模型与动作预设分享。
- macOS / Linux 发布支持。
- 完整场景对象控制。
- 长会话 FTS、搜索与压缩。
- 插件市场或通用外部 Memory Provider。

## 7. 不再推进

- 一次性整体重写。
- 旧 XML / 文本内嵌标签式 AI 指令。
- 旧 `vrm_segment / vrm_complete` 输出链。
- 旧 `/messages` SSE 主链。
- 把历史“运行时升级方案”作为独立实施蓝图。
