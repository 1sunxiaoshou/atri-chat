# ATRI vNext：一次做通、推倒与重新开始

这是一份已删除方案的共同复盘，不描述当前 Python、FastAPI、React 与 Tauri 实现。它既记录项目负责人的产品判断，也保留协作开发过程中形成的工程判断。前者关心用户是否真正看见、理解并完成任务，后者关心范围、复杂度和维护成本怎样逐渐偏离产品价值。两种视角共同解释了为什么一条技术上跑通的链路仍然需要被放弃。

当前实现以 [系统设计](../02-架构/系统设计.md) 为准；已经决定推进的工作以 [路线图](./路线图.md) 为准。本文只用于保留历史背景和可复用的产品、工程经验。

## 结论

ATRI vNext 曾在很短时间内做通一条完整的 Windows 桌面 AI 伴侣技术链路：桌面 UI、对话、数据持久化、Agent 工具、用户审批、Unity 角色表现、开发包和故障恢复都曾有可运行实现。

我们最终清除了其中绝大部分代码。原因不是它完全不能运行，而是开发顺序逐渐倒置：原本希望先由前端定义产品的样子和交互，再由后端实现这些需求；实际却是后端持续更新和增加能力，前端始终追赶，最终无法把已经存在的功能组织成一致、好用的用户体验。

与此同时，.NET 与 Avalonia 这套桌面 UI 技术栈缺少成熟的 AI 前端组件和交互框架。对话流、工具审批、模型配置、角色管理和运行状态都需要大量自行设计与实现，开发成本很高，生成的图标和视觉资源也始终没有达到预期。Microsoft Agent Framework 本身又没有表现出足以抵消长期维护成本的突出价值。

这次清理不是否定所有工作，而是停止让已经投入的工程成本替未来做决定。

## 我们原本想做什么

目标一直很简单：做一个本地优先、可以长期陪伴用户的桌面 AI 角色。它应当能够对话、记住用户可见的历史、完成经过授权的任务，并用表情、动作和语音呈现角色感。

旧项目留下了许多技术和目录结构。vNext 决定不直接继承它们，而是从产品边界重新开始：只保留一个生产 Agent Runtime、一个产品事实源，以及一个可独立替换的角色渲染端。

## 我们实际做过什么

2026 年 7 月 18 日，仓库依次完成了以下纵切：

1. 建立产品原则和旧资产筛选规则。
2. 建立 .NET 10 与 Avalonia 12 桌面基础。
3. 用 SQLite 和 EF Core 保存角色、会话与消息。
4. 定义桌面与 Unity 之间的 v1 表现协议。
5. 接入 Microsoft Agent Framework 和 OpenAI-compatible Provider，实现流式对话。
6. 建立独立 Unity Player，加载 VRM 1.0 角色并执行表情、挥手和 WAV 播放。
7. 将产品 run 与 Unity 表现关联，处理取消、超时、故障和降级。
8. 加入本地工具、一次性审批和单个可选 MCP server。
9. 生成可校验的 Windows 开发包，并补齐正常关闭与重启恢复。

在被删除前，这条链路能够完成：配置 Provider、创建角色、流式对话、提出工具调用、允许一次或拒绝、继续同一个 Agent run、触发表情和动作、取消生成，并在重启后恢复会话。

这些能力不是只写在设计稿里。协议、数据库、恢复路径、Unity Player 和开发包都有对应测试或 smoke 验证。

但“代码存在”和“产品能力存在”不是同一件事。只有当一个功能拥有可理解的视觉表达、完整的操作入口、明确的过程反馈，并能让用户走完整条使用链路，它才真正成为产品的一部分。

如果用户看不见一个功能、不知道如何触发、无法理解当前状态，或者不能完成从开始到结果的全过程，那么从用户角度看，它与尚未实现几乎没有区别。

## 做对了什么

一些原则经受住了这次清理，仍值得保留：

- 用户可见的消息和运行状态应属于产品，而不是被某个 Agent 框架的私有 session 垄断。
- 高权限工具应逐次展示具体目标和影响，批准不能自动扩大成永久授权。
- Unity 只负责表现。它可以崩溃、重启或被替换，但不能决定对话与数据事实。
- 密钥不应进入日志、消息、模型上下文或普通配置。
- 不用静音文件、提示音或假动画冒充尚未完成的 TTS 与角色能力。
- 依赖、模型资产、许可证、版本和哈希应当可追溯。

这些工作说明我们能够把复杂边界做得可靠。问题不在工程是否认真，而在认真用得太早、太广。

## 哪里失败了

### 1. 我们把技术完成误当成了产品完成

开发过程中很容易形成一种错觉：数据库已经有表、Runtime 已经能调用、协议已经跑通、测试已经通过，于是脑海中会把这些能力标记为“做完了”。

但在正式投入使用前，许多问题不会自然暴露。只有用户实际走完整条链路，才会发现入口是否明显、概念是否能理解、状态是否有反馈、失败后能否继续，以及模块之间是否真的连得起来。没有完成这种端到端体验验证时，我们不仅高估了已经做完的部分，也没有意识到还有多少必要工作根本没有被想到。

视觉表现并不是最后附加的一层装饰。对于桌面 AI 伴侣，它本身就是功能被感知、角色被理解、状态被信任的主要媒介。视觉和交互没有建立起来，后端能力就无法转化成用户价值。

### 2. 后端始终走在前端前面

最初的开发设想是“前端驱动后端”：先把用户看到的页面、操作路径和反馈状态搭出来，确认每个模块应该怎样使用，再让后端逐项实现真实能力。

实际过程正好相反。后端不断加入持久化、流式对话、Runtime 投影、工具调用、审批、MCP、Unity 表现、恢复和打包，希望先让整条链路跑通。前端没有同步形成足以承载这些能力的交互模型，只能不断为已经存在的后端功能补入口、表单、状态和错误反馈。

这种倒置造成了大量不一致：后端已经定义了完整状态和边界，前端却还没有决定用户应该怎样理解和操作它们。功能越多，前端需要补上的交互债务越大。

### 3. 前端本身不足以支撑已经实现的功能

ATRI 不只是一个聊天框。角色、会话、模型服务、能力连接、工具审批、运行状态、错误恢复和设置管理都需要清晰的层级与反馈。我们没有先为每一个模块想清楚它在产品中应该以什么形式出现，也没有形成一套完整的管理与配置体验。

结果是功能在后端存在，但用户很难自然地发现、理解和使用。基础交互体验尚未稳定，更复杂的 Agent 能力只会继续放大问题。

### 4. 当前 UI 技术栈让前端成本过高

.NET 与 Avalonia 能够构建桌面应用，但在面向 AI 产品的现成 UI 生态上较弱。项目没有找到足够成熟的对话线程、流式消息、工具调用卡片、审批流程、模型配置和 Agent 状态组件，只能从基础控件开始自行组合。

这不仅增加代码量，也把大量时间消耗在控件状态、布局、自适应、动画、可访问性和视觉细节上。生成或改造的交互图标也没有达到预期，进一步说明“先快速搭出满意前端，再由后端跟进”的路径在当前技术栈下并没有实现。

前端开发成本持续上升，同时仍然影响最终的设计一致性和操作体验。这已经不是补几个页面就能解决的问题，而是技术栈、组件生态与产品形态之间的错配。

### 5. Microsoft Agent Framework 没有提供足够的独特价值

选择 Microsoft Agent Framework，是希望借助官方框架减少 Agent loop、流式执行、工具调用和 session 管理的重复建设。但在实际纵切里，产品仍然需要自行处理消息事实、投影恢复、审批状态、工具安全、错误归一化和 UI 映射。

框架并没有显著减少产品最昂贵的部分，反而增加了版本、投影格式、上下文重建和长期兼容维护。考虑到它没有表现出不可替代的突出能力，继续围绕它维护产品边界并不划算。

### 6. Unity 证明了可行性，没有解决产品体验

Unity renderer 能加载 VRM 1.0、播放 `happy`、执行 `wave`、校验 WAV、接受取消并返回结构化反馈。这证明了技术路线可行。

但它也带来了第二个进程、固定 Unity 版本、较重的冷导入和构建成本、命名管道、额外打包体积及独立故障路径。当前角色只是官方示例 `Seed-san`，画面是深色背景、固定相机和原始模型姿态，没有透明桌宠、交互控件、口型、动画混合或成熟构图。它更像协议夹具，而不是 ATRI 的最终角色体验。

在没有产品测量前，这些成本还不能证明不可替代。

### 7. “开发包可运行”离“产品可用”仍很远

项目生成过自包含 Windows 开发包，但没有完成正式安装器、代码签名、更新源、升级、回滚和公开发布验证。语音协议能够播放真实 WAV，却没有选定真实 TTS，也没有完成会话到语音的产品链路。

更重要的是，即使这些发布工程全部补齐，也不会自动改善前端交互。技术演示和完整链路不能被等同为一个用户愿意使用的产品。

### 8. 文档和实现开始互相绑架

随着决策增加，文档越来越像在描述一套已经确定的长期架构。清理代码后，其中仍有不少“当前实现”的文字指向已删除目录。这说明文档不再只是帮助思考，也开始让临时方案看起来比实际更永久。

## 为什么选择清除，而不是继续修补

继续保留旧桌面纵切会产生几种偏见：

1. 因为代码已经存在，未来需求会被迫适配 Avalonia、MAF、SQLite 和现有 schema。
2. 前端会继续围绕已经实现的后端状态补界面，而不是先决定正确的产品交互。
3. 因为测试很多，删除错误方向会显得比增加更多兼容层更昂贵。
4. 因为链路能运行，团队容易继续优化技术完成度，而不是解决用户实际面对的体验问题。

从产品视角看，最直接的问题是视觉与交互没有把功能变成用户能够感知的能力；从工程视角看，更深一层的问题是我们过早把未经体验验证的能力固化成数据库、协议、恢复语义和测试合同。前者让产品不可用，后者让纠正方向越来越昂贵。

所以我们选择完整删除桌面应用、产品内核、SQLite、Agent Runtime、MCP、测试项目、解决方案和开发包实现，而不是继续让前端追赶一套已经膨胀的后端合同。Microsoft Agent Framework 也随这条实现一并放弃，不再因为它是微软框架或代码已经接入，就默认它应当成为下一次方案的基础。

这是一次范围和技术路线的止损：放弃的是前后端失衡的实现方式、成本过高的 UI 路径和价值不足的 Runtime 选择，不是桌面 AI 伴侣这个目标。

## 留给下一次的原则

如果 ATRI 再次开始建设桌面实现，应当从一个用户能直接感知的纵切开始，而不是恢复被删除的目录。

下一次首先需要回答：

- ATRI 在桌面上的最小存在形式是什么？
- 哪一个高频场景足以让用户第二天再次打开它？
- 每一个页面、管理模块和配置项，用户应该怎样发现、理解并完成操作？
- 能否先用真实前端原型验证完整交互，再决定后端状态与接口？
- 一个功能是否已经有入口、过程反馈、失败恢复和可验证的最终结果，而不只是后端实现？
- 用户能否在没有开发者解释的情况下走完整条链路？
- 新技术栈是否拥有足够成熟的 AI UI 组件，能够降低而不是转移前端成本？
- 没有 Agent 工具、MCP 和复杂恢复时，基础角色与对话体验是否已经成立？
- Unity 相比更轻的渲染方案提供了什么经过测量、不可替代的价值？
- Agent Runtime 相比直接使用模型 API 提供了什么不可替代的价值？
- 新增的每个依赖、进程、协议和事实源，是否服务于已经确认的需求？

下一次应当真正让前端体验驱动开发：先确定用户看到什么、怎样操作、每一步获得什么反馈，并用可操作原型走通完整链路，再实现支撑这些交互所必需的最小后端。功能的完成标准应是用户能够感知并成功使用，而不是代码、接口或测试单独存在。只有出现第二个真实需求时，再抽象；只有产品数据证明必要时，再扩大系统。

## 最后

这段历程最重要的成果不是某段代码或某项技术，而是几条更清楚的判断：技术链路跑通，不等于产品成立；后端实现存在，不等于用户拥有这项能力；视觉和交互不是功能完成后的包装，而是功能成为产品的必要条件；能够继续堆功能，也不等于应该继续。

删除已经完成的工作并不可惜。真正可惜的是因为舍不得已经投入的成本，让一个未经验证的实现继续替产品做决定。
