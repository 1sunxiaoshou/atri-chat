# 架构审计与重构建议

本文记录一次 ATRI Chat 实现的架构审计结果。它不是一次性重写蓝图，也不是已经批准的实施路线，而是基于审计时主链给出的风险清单、目标边界和渐进迁移建议。已经决定推进的事项以 [路线图](./路线图.md) 为准。

审计日期：2026-07-17。

> [!NOTE]
> 本文中的文件位置、代码规模、测试数量和问题状态均为审计日期当时的快照。实施任何建议前，应重新核对当前代码与 [系统设计](../02-架构/系统设计.md)，已修复事项应从路线图移除或标记完成，而不是继续把本文当作实时问题清单。

## 1. 结论摘要

ATRI Chat 不是一个“完全没有架构”的项目。当前实现已经具备以下正确方向：

- Tauri 负责桌面运行时目录、动态端口和 Python sidecar 生命周期。
- React、FastAPI、Agent、SQLite、VRM 渲染之间已有清晰的主链。
- 聊天已经统一到 `/agent-stream`，旧文本标签和旧 SSE 链路已退出主线。
- Agent 使用单编译图、动态模型、动态提示词、动态工具过滤。
- ASR、TTS、VRM 都有独立模块，VRM 高频更新主要留在 `useFrame`，没有通过 React state 逐帧渲染。
- 代码格式、Python 静态检查、TypeScript 类型检查和现有自动化测试整体可通过。

真正的问题不是“目录完全混乱”，而是系统正处于重构中间态：文档中的分层已经出现，但很多运行时一致性仍依赖调用者自觉维持。最严重的风险集中在四处：

1. **本地后端没有形成可信边界。** 后端监听全部网卡、允许任意 Origin、没有请求鉴权，同时配置接口会返回明文 API Key。
2. **会话存在多个事实源。** 业务消息表、LangGraph checkpoint、LangGraph store、Markdown 记忆分别持久化，却没有统一的创建、清空、删除和恢复语义。
3. **VRM 只有执行函数，没有真正的 Director。** 单批命令内部看似顺序执行，但不同批次可以并发，取消、抢占、动作完成、音频结束和反馈关联都不完整。
4. **异步接口中混入大量同步 I/O。** 同步 SQLAlchemy、同步模型发现、同步文件扫描和部分大文件读写会阻塞事件循环；单用户时不一定立即暴露，但会放大流式聊天、TTS 和管理操作互相影响的问题。

因此，不建议更换整个技术栈，也不建议重写。优先级应当是：

```text
安全边界
 -> 会话一致性
 -> 实时事件协议与 VRM Director
 -> 数据访问和异步边界
 -> 渲染与缓存性能
 -> 模块目录整理
```

## 3. 当前系统结构

### 3.1 进程结构

```text
Tauri 主进程
 ├─ 解析 installed / portable / development 运行时目录
 ├─ 分配动态端口
 ├─ 启动 Python sidecar
 └─ 等待 GET /api/v1/health

React WebView
 ├─ 通过 Tauri command 获取后端端口
 ├─ HTTP CRUD / 文件上传
 ├─ POST + SSE 接收 Agent stream
 └─ React Three Fiber 渲染 VRM

FastAPI sidecar
 ├─ 业务 SQLite: app.db
 ├─ LangGraph checkpoint: checkpoints.db
 ├─ LangGraph store: store.db
 ├─ Markdown 角色记忆: data/memory/{character_id}
 └─ VRM / motion / TTS / ASR 文件资源
```

### 3.2 后端结构

名义上的分层是：

```text
api/routes
 -> core/services
 -> core/repositories
 -> core/db
```

但实际代码同时存在三种路径：

```text
route -> ORM
route -> service -> repository -> ORM
agent middleware -> service -> repository -> ORM
```

大部分 CRUD 路由直接操作 ORM 和事务，只有模型同步、Agent 模型解析和消息持久化部分使用 service / repository。因此当前并不是稳定的分层架构，而是两套数据访问风格并存。

### 3.3 前端结构

前端目前混合使用：

- `App.tsx` 本地 state：视图、会话、当前选择、侧边栏、Toast、临时模型。
- Zustand：核心列表缓存、聊天输入、音频设置、VRM 配置和运行态。
- Hook 内部 state：Agent stream、TTS、ASR、VRM 编排。
- 组件 props：在 `App -> ChatInterface -> ChatHeader / RightSidebar / VRM` 之间传递活动对象和回调。

这种组合对当前规模仍可工作，但没有明确区分：

- 服务端数据缓存
- 会话级 UI 状态
- 持久化用户设置
- 高频渲染状态
- 一次 Agent run 的临时状态

结果是相同概念可能同时出现在 React state、ref、Zustand 和后端 runtime registry 中。

## 4. 双向传输与 VRM 渲染是如何工作的

### 4.1 上行：用户输入到 Agent

```text
ChatInput
 -> ChatInterface.handleSend
 -> useAgentStream.sendMessage
 -> @langchain/react FetchStreamTransport
 -> POST /api/v1/agent-stream
 -> AgentCoordinator.stream_runtime_events
 -> LangGraph agent.astream
```

前端先插入乐观 `HumanMessage`，再把角色、模型、显示模式和推理参数放入 request context。后端使用 `conversation_id` 作为 LangGraph `thread_id`。

### 4.2 下行：模型 token 和 custom event

后端把 LangGraph v2 stream part 转成 SSE：

```text
messages / updates / custom / metadata
 -> agent_stream._to_sse
 -> FetchStreamTransport
 -> useStream
 -> useAgentStream
```

文本 token 进入消息状态；推理内容被转换为 `reasoning` custom event。

### 4.3 VRM 命令链

模型在 VRM 模式下可以调用：

- `perform_actions(commands: string[])`
- `control_camera(action, preset, durationMs?)`

链路如下：

```text
LLM tool call
 -> Python 解析字符串命令
 -> say 命令逐段执行 TTS
 -> ToolRuntime.stream_writer 写 custom event
 -> SSE: vrm.perform_actions / vrm.control_camera
 -> useAgentStream.onCustomEvent
 -> ChatInterface 回调
 -> useVRM.executeCommands / executeCameraCommand
 -> Zustand VRM runtime
 -> Character / GenshinControls / useFrame
```

执行完成后，前端另外发送：

```text
POST /api/v1/runtime/vrm-feedback
```

因此这里并不是真正的单连接全双工传输，而是：

- 一个长连接 HTTP response 负责 SSE 下行；
- 普通 HTTP POST 负责用户输入和执行反馈上行。

对单用户桌面应用而言，这个选择是合理的。当前没有必要仅为“看起来更实时”改成 WebSocket。只有需要服务端主动推送、多设备、实时打断、二进制音频复用或同一 run 上的高频双向控制时，WebSocket 才会产生明显收益。

### 4.4 渲染链

```text
useVRMStore.runtime
 -> ChatVRMViewerR3F
 -> VRMCanvas
 -> AIStage
 -> Character
 -> useVRMLoader / useMotionController / useLipSync / blink / lookAt
 -> 单一 useFrame 中更新 vrm、mixer、口型、眨眼和视线
```

高频更新不经过 React render，这一点设计正确。主要性能问题来自 WebGL 配置、资源生命周期、永久逐帧工作和业务编排竞态，而不是 React Three Fiber 本身。

## 5. 技术选型评价

### 5.1 Tauri + Python sidecar

结论：**适合项目能力目标，但不能再宣传为纯粹“极致轻量”。**

优点：

- Tauri 比 Electron 壳层更轻。
- Python 对 LangChain、ONNX、音频和模型生态友好。
- sidecar 隔离了 WebView 和 AI 依赖，开发效率高。

代价：

- 实际产品包含 Rust、WebView、Python 解释器、PyInstaller 产物和大量 AI 依赖。
- 启动、日志、端口、退出、崩溃恢复和打包都变成跨进程问题。
- Python 后端一旦暴露错误，就不仅是“本地内部实现”，而是一个真实网络服务。

建议保留该组合，但把“sidecar 是可信本地服务”的安全契约补完整。

### 5.2 FastAPI + SQLAlchemy + SQLite

结论：**选型合理，实现方式需要统一。**

SQLite 很适合单用户桌面应用，WAL、外键和 Alembic 也已经启用。问题在于 65 个路由几乎都声明为 `async def`，却大量执行同步 ORM、同步文件操作和同步远端模型发现。

可选方案只有两种，应选一种并贯彻：

1. 简单 CRUD 改为同步 `def`，让 FastAPI 放到线程池；流式、TTS、ASR 保持异步。
2. 全面迁移 SQLAlchemy `AsyncSession`，并确保 Provider SDK 和文件操作也有清晰的异步边界。

对当前单机项目，第一种方案成本更低；没有必要为了“全异步”做大迁移。

### 5.3 LangChain / LangGraph

结论：**有使用理由，但职责过宽。**

项目需要模型切换、工具调用、checkpoint 和流式事件，LangGraph 并非完全多余。问题是它同时承担运行状态、短期历史和工具循环，而业务数据库又保存另一份消息历史。

建议：

- LangGraph 继续作为 Agent 执行引擎。
- 业务消息 / run event 成为产品事实源。
- checkpoint 明确为可重建的执行投影，或者反过来彻底放弃业务消息副本；不能继续模糊并存。
- 不要让 LangGraph 类型直接扩散到所有前端和业务 API。

### 5.4 React + Zustand + React Three Fiber

结论：**整体合适。**

- React 负责界面和业务交互。
- Zustand 适合 VRM 配置和低耦合全局状态。
- R3F 让 VRM 生命周期与 React 组件生命周期能被统一管理。

不合理之处在于 Zustand 被同时用作用户设置、渲染状态和命令执行状态，而服务端数据又自行实现缓存。建议按状态类型拆边界，而不是盲目换状态库。

### 5.5 SSE

结论：**保留。**

当前通信量以服务端下行为主，SSE 简单、可调试、兼容 LangGraph stream。首要任务是完善事件版本、run ID、取消语义和幂等，而不是更换协议。

### 5.6 Markdown 长期记忆

结论：**适合 MVP，但不是可扩展检索系统。**

优点是透明、可编辑、角色隔离、实现成本低。当前搜索会遍历所有 Markdown 并逐文件读取，写入没有锁、大小限制和原子替换。并发 run 或长期积累后会遇到性能和数据竞争。

近期可继续保留 Markdown，但要增加文件大小、目录总量、原子写和每角色写锁。只有出现大量记忆和语义检索需求后，再引入 FTS 或向量检索。

## 6. 分级问题清单

### 6.1 P0：必须先修复

#### P0-1 本地后端暴露到局域网，且没有鉴权

证据：

- `core/bootstrap.py` 使用 `host="0.0.0.0"`。
- CORS 配置为 `allow_origins=["*"]`、任意 method 和任意 header。
- API 没有 session token 或本地客户端签名。
- `api/routes/providers.py` 和 `api/routes/tts_providers.py` 会返回完整 `config_payload`。
- Provider schema 已将 `api_key` 标记为 sensitive，但返回路径没有脱敏。
- `frontend/src-tauri/tauri.conf.json` 中 CSP 为 `null`。

影响：

- 同一局域网设备可以访问后端端口。
- 恶意网页可扫描 localhost / LAN 服务并尝试调用配置、上传和删除接口。
- LLM / TTS API Key 以明文 JSON 存在 SQLite，并会被接口返回。
- 头像、VRM、动作上传缺少统一大小限制，可造成内存或磁盘耗尽。

建议：

1. 后端只监听 `127.0.0.1`。
2. Tauri 每次启动生成高熵 session token，同时注入 Python 和前端；所有 `/api` 请求必须携带。
3. CORS 只允许实际 Tauri / 开发 Origin。
4. 列表和详情接口只返回 `has_api_key` 或脱敏值；更新时空值表示保留旧密钥。
5. Windows 发行版优先用 Credential Manager / DPAPI 保存密钥，SQLite 只保存引用或密文。
6. CSP 使用最小允许列表。
7. 所有上传统一做流式大小限制、扩展名和内容签名验证。

#### P0-2 会话清空 / 删除没有同步清理 LangGraph checkpoint

证据：

- `conversation_id == thread_id`。
- Agent 每轮从 `agent.aget_state()` 读取 checkpoint 历史。
- `DELETE /conversations/{id}` 和 `DELETE /conversations/{id}/messages` 只操作 `app.db`。
- 当前代码没有调用 checkpointer 的 thread 删除能力。

影响：

- 用户界面显示“已清空”，下一轮 Agent 仍可能携带旧历史。
- 删除会话后，checkpoint 数据成为孤儿，磁盘持续增长。
- 业务消息和 Agent 记忆发生不可见分叉，属于隐私和产品语义错误。

建议：

- 先定义唯一语义：清空消息是否保留长期记忆、是否保留标题、是否保留 run event。
- 由一个 `ConversationLifecycleService` 统一执行数据库消息、checkpoint、run event 和临时 TTS 的清理。
- 增加集成测试：清空后下一轮模型上下文不得出现旧消息。
- 如果跨数据库无法原子提交，使用可恢复的 outbox / cleanup job，并把清理状态记录在业务库。

### 6.2 P1：高优先级设计缺陷

#### P1-1 VRM 没有真正的命令队列、取消和抢占

当前 `executeCommands()` 只保证一个数组内部使用 `for` 顺序遍历，但每个 `vrm.perform_actions` custom event 都会在 `ChatInterface` 中独立启动 Promise。多个工具调用可以并发执行，并共同操作同一个 `HTMLAudioElement` 和同一个 Zustand runtime。

具体问题：

- 新 run 的 `startThinking()` 会 `pause()` 旧音频，但旧 `playSpeechAudio()` Promise 不会收到 `ended`，可能永久悬挂。
- 新的音频播放会覆盖旧音频的 `onended` / `onerror`，旧批次无法完成。
- `motion` 命令只设置 `motionUrl`，不会等待动作结束就执行下一条命令。
- `isWaitingForMotionRef` 和 `motionFinishedCallbackRef` 存在，但没有被命令执行路径设置。
- 回复动作结束后没有可靠回到 idle，容易停在最后一帧。
- 反馈没有把 `toolCallId` / `run_id` 作为关联主键，后端只保存全局最近一次 VRM feedback。

建议新增前端 `VRMDirector`：

```text
SSE event
 -> validate
 -> enqueue(run_id, batch_id)
 -> execute one command
 -> await completion / timeout / cancellation
 -> emit correlated feedback
 -> advance queue
```

Director 至少需要：

- 单消费者队列
- `AbortController` 或 run generation token
- 新消息抢占策略
- motion / speech / wait 的明确完成条件
- 每条命令超时
- idle / thinking / speaking / acting 状态机
- 按 `run_id + tool_call_id + command_id` 反馈

#### P1-2 命令协议在 Python 和 TypeScript 中重复解析

`emotion happy`、`motion xxx`、`say happy | text`、`wait 300` 在后端和前端各有一套解析器。虽然现在有一致性脚本，但每次新增转义、可选字段或国际文本规则，都必须同步两个实现。

建议把工具入参改为结构化判别联合：

```json
{
  "version": 1,
  "commands": [
    { "id": "c1", "type": "emotion", "expression": "happy" },
    { "id": "c2", "type": "say", "expression": "happy", "text": "欢迎回来" },
    { "id": "c3", "type": "wait", "duration_ms": 300 }
  ]
}
```

后端只做 schema 校验，前端只消费类型化结果。TypeScript 类型可从 JSON Schema / OpenAPI 生成，避免继续维护双解析器。

#### P1-3 TTS 被放在 Agent 工具关键路径中串行执行

`perform_actions` 在写出 custom event 前，会遍历全部 `say` 命令并依次等待 TTS。结果是：

```text
模型已决定动作
 -> 等待第 1 段 TTS
 -> 等待第 2 段 TTS
 -> ...
 -> 前端最终才看到动作事件
```

影响：动作首帧延迟等于所有语音合成延迟之和；TTS 失败虽然被吞掉，但仍消耗超时窗口。

建议：

- custom event 先发送动作意图。
- `say` 到达 Director 时按需请求 TTS，或者后端并发生成并发送 `speech.ready` 事件。
- TTS 缓存应有 single-flight，避免相同 key 并发重复合成。
- 临时文件使用原子 rename，并配置大小 / 时间淘汰。

#### P1-4 TTS 缓存和实例缓存语义错误

问题包括：

- 前端 `audioCache` 只以文本为 key，不包含角色、音色、供应商、语言和配置版本。同一句话切换角色后可能播放上一个角色的声音。
- 角色 TTS 把合并后的 config 显式传入 `TTSFactory.create_tts()`；该分支被定义为“测试连接临时实例，不缓存”，因此角色语音每次都新建 Provider 实例。
- 后端磁盘 TTS 缓存没有淘汰策略。

建议统一缓存键：

```text
provider_id + provider_version
+ voice_id + voice_version
+ language
+ normalized_text
```

Provider client 缓存和音频内容缓存应分开管理。

#### P1-5 业务消息和 checkpoint 双写，不具备 turn 原子性

一轮请求会：

1. 在 `app.db` 保存用户消息并 commit。
2. LangGraph 在 `checkpoints.db` 更新状态。
3. `persist_agent_messages` 遍历最终 state，把 AI / tool 消息逐条 commit 到 `app.db`。

任一步骤失败都可能产生半轮数据。当前异常通常只写日志，stream 仍可能完成；也没有 `turn.status` 表示 pending、completed 或 failed。

建议：

- 增加 `agent_runs` / `turns`，记录 run 状态、开始结束时间、模型和错误。
- 消息批量落库，一轮使用明确事务边界。
- 使用 `conversation_id + lc_message_id` 幂等键是正确方向，应继续保留。
- checkpoint 明确为投影，提供重建和清理工具。

#### P1-6 长会话查询会产生错误结果和过量加载

问题包括：

- 会话列表通过 `selectinload(Conversation.messages)` 加载每个会话的全部消息，只为计算 `len(messages)`。
- 消息接口固定按时间正序取前 100 条；超过 100 条后，前端加载的是最旧消息，而 checkpoint 仍包含新消息。
- 前端没有 cursor / load-more。

建议：

- 会话列表使用 `COUNT(*)` 子查询或维护投影计数。
- 消息列表按 `(created_at, id)` cursor 分页，初次加载最新 N 条，再在前端恢复正序。
- 对“业务表消息”和“Agent 上下文消息”增加一致性测试。

#### P1-7 同步 I/O 阻塞异步服务

已确认的路径：

- 65 个路由全部为异步风格，大量使用同步 SQLAlchemy Session。
- 供应商模型发现使用同步 OpenAI client 或 `requests.get()`，直接在 async route 中执行。
- LangGraph store 的 `abatch()` 直接调用同步 `batch()`。
- Markdown 记忆工具同步扫描和读取文件。
- VRM / motion 上传使用 `await file.read()` 后一次性写入。

建议先做低成本修正：

- 纯 CRUD 路由改为同步 `def`。
- 远端模型发现放入 `asyncio.to_thread()`，或统一改用异步 client。
- 大文件上传分块写入并限制大小。
- memory 工具放线程池；写入加每角色锁和原子替换。

#### P1-8 VRM WebGL 和资源生命周期存在性能 / 正确性风险

主要问题：

- `preserveDrawingBuffer: true` 对所有 VRM Canvas 永久开启，只为了偶发截图，会增加显存和合成成本。
- 所有 VRM mesh 被设置为 `frustumCulled = false`，即使离开视野仍提交渲染。
- `useGLTF` 自带缓存，但组件卸载时又 `deepDispose` 缓存中的场景；缓存可能保留已释放对象。
- 清理时使用的原始 URL 和实际绝对 URL 可能不同，`useGLTF.clear()` 不一定命中。
- Canvas 默认持续渲染；没有窗口隐藏、页面不可见或低性能设备降帧策略。
- 相机控制每帧重新创建 preset Vector3，并在 preset 状态不清除时不断重设目标；用户手动 Orbit 也可能与命令目标争抢。
- 地面碰撞每帧创建多个 Vector3，可改为复用 ref。

建议：

1. 普通聊天 Canvas 关闭 `preserveDrawingBuffer`；截图时使用独立 render target 或短时策略。
2. 只对确有包围盒问题的对象关闭剔除。
3. 资源缓存和 dispose 必须由同一所有者管理。
4. 统一 canonical resource URL 作为缓存 key。
5. 增加可配置 DPR / FPS，窗口不可见时暂停或降低帧率。
6. 相机命令进入 Director，结束后清除目标状态；逐帧向量复用。

#### P1-9 ASR 单例缺少并发保护

`SenseVoiceASR` 是全局单例，语言或 INT8 参数改变时会替换 recognizer。初始化和 `decode_stream` 没有锁。两个并发请求可能互相切换 recognizer，底层 C++ 对象也未必支持并发调用。

单用户 UI 中概率较低，但局域网暴露或重复点击会放大问题。建议使用串行 worker / `asyncio.Lock`，并将不同配置的 recognizer 缓存为有限实例，而不是修改单一共享实例。

### 6.3 P2：中期可维护性问题

#### P2-1 路由承担过多业务和事务逻辑

`characters.py`、`providers.py`、`conversations.py` 等路由包含查询、校验、文件删除、事务、响应转换和业务规则。另一方面 Repository 自己 commit，导致 service 无法组合原子事务。

建议采用轻量 application service：

```text
route: HTTP 解析与状态码
application service: 用例、权限、事务
repository: 查询和持久化，不自行 commit
adapter: LLM / TTS / ASR / filesystem / LangGraph
```

不需要引入完整 DDD；只要让事务边界属于用例，而不是 Repository。

#### P2-2 schema 和类型没有单一来源

- `ResponseModel` 在 `api/schemas.py` 和 `api/schemas_runtime.py` 重复。
- Character、Conversation、TTS 等 schema 一部分集中定义，一部分定义在 route 文件。
- 前端 TTS Provider / Voice Asset 在多个组件重复声明。
- `AgentStreamCustomEvent` 只有开放字符串索引，无法穷尽检查事件类型。
- `Conversation.id` 和兼容字段 `conversation_id` 类型不一致。

建议：

- 后端按 feature 组织 request / response schema。
- 删除已退出主线的兼容字段。
- 从 OpenAPI / JSON Schema 生成前端 API DTO。
- 内部 UI model 可以独立，但必须通过 adapter 从 DTO 显式转换。

#### P2-3 数据模型存在可漂移冗余

`Character` 同时保存 `primary_model_id` 和 `primary_provider_config_id`，而 `Model` 已经保存 `provider_config_id`。创建和更新角色只验证 model 是否存在，没有强制两个 provider ID 一致。

建议只保存 `primary_model_id`，供应商通过 Model 关系获取。若保留冗余作为投影，则必须用数据库约束或单一写入服务维护。

#### P2-4 依赖面过宽

所有 LangChain Provider、ASR、SciPy 等都在主依赖中。即使用户只用一个 OpenAI-compatible Provider，发行包仍携带多套适配器。

建议：

- 内置保留最常用的 OpenAI-compatible、Ollama 和一个原生 Provider。
- 其他 Provider 变成可选 extras 或构建 profile。
- 运行时 registry 可以保留，但避免“安装即携带全部 SDK”。

#### P2-5 CapabilityRegistry 超前抽象

`runtime_status.py` 用较多代码抽象 manifest、warmup、reset、event 和 frontend feedback，但当前只有四种能力，大部分没有 shutdown，前端主要在右侧栏每 15 秒轮询状态。

这套抽象不是错误，但收益暂时低于复杂度。建议在新增真实生命周期需求前冻结功能，不继续扩张“控制平面”；优先让已有状态准确、可关联、可测试。

#### P2-6 后台线程动态注册路由

Deferred route 在应用开始提供服务后，通过后台线程调用 `include_router()`。这会产生短暂 404 窗口，也依赖运行时修改路由表的线程安全。

路由模块本身不应承担昂贵初始化。建议所有路由同步注册，真正重型 Provider / ASR / TTS 实例继续惰性创建。

#### P2-7 状态管理边界模糊

`App.tsx` 和 `ChatInterface.tsx` 仍承担较多协调职责，`useVRM.ts` 同时管理模型加载、动作绑定、idle、thinking、speech、字幕、camera 和反馈结果。

建议按职责拆分，而不是按文件大小拆分：

- `useConversationSelection`
- `useAgentRun`
- `VRMDirector`
- `VRMRenderStore`
- `VRMAssetRepository`
- `SpeechPlaybackService`

#### P2-8 缺少运行取消和失败恢复

前端没有显式 Stop / Abort；后端 stream 错误通过 custom event 发送，但 HTTP 仍可能是 200。用户无法可靠区分：模型失败、TTS 失败、VRM 部分失败、反馈失败或网络中断。

建议引入统一 run 状态和错误码，并支持前端 abort。失败后保留用户消息没有问题，但必须标记该 turn failed，并允许重试且不重复持久化。

### 6.4 P3：清理项

- 项目版本是 `1.0.1`，FastAPI 元数据硬编码为 `1.1.0`。
- Python 包元数据声明 `Operating System :: OS Independent`，实际发行和文档明确为 Windows 优先。
- `AgentCoordinator.start_services()` / `stop_services()` 是空实现。
- i18n 当前有两处硬编码文案；独立运行检查会失败。
- 发布预检只检查版本、锁文件和资源，不包含 ruff、pytest、type-check、lint、i18n。
- 前端存在较多重复 `any`、重复 DTO 和散落 console 日志。
- 部分注释宣称“缓存”“按权重随机”“完全隔离”，实际行为与描述不完全一致。

## 8. 建议的目标架构

目标仍然是一个模块化单体，不增加微服务。

```text
Tauri Runtime Boundary
 ├─ loopback only
 ├─ session token
 └─ process / data-root lifecycle

FastAPI Transport
 ├─ thin routes
 ├─ typed request / response / stream events
 └─ auth + error middleware

Application Use Cases
 ├─ ConversationLifecycleService
 ├─ AgentRunService
 ├─ CharacterAssetService
 ├─ ProviderConfigService
 └─ SpeechService

Ports / Adapters
 ├─ LangGraphAgentAdapter
 ├─ SQLAlchemy repositories
 ├─ CheckpointRepository
 ├─ SecretStore
 ├─ TTS / ASR providers
 └─ AssetStorage

React
 ├─ server data cache
 ├─ Agent run controller
 ├─ typed stream adapter
 ├─ VRM Director
 └─ pure R3F render adapter
```

### 8.1 推荐的 feature 边界

不必立即移动所有文件。新代码可以先按下列边界收口：

```text
core/chat/
  application.py
  runs.py
  events.py
  checkpoints.py

core/characters/
  application.py
  repository.py
  assets.py

core/providers/
  application.py
  secrets.py
  adapters/

frontend/features/chat/
frontend/features/vrm-director/
frontend/features/characters/
frontend/rendering/vrm/
```

迁移过程中保留原路径 facade，避免一次性修改全部 import。

### 8.2 实时事件协议

建议所有事件至少包含：

```text
protocol_version
conversation_id
run_id
turn_id
sequence
event_type
payload
```

核心事件：

- `run.started`
- `message.delta`
- `reasoning.delta`
- `action.batch`
- `speech.ready`
- `usage.completed`
- `run.completed`
- `run.failed`

前端反馈：

- `action.started`
- `action.completed`
- `action.failed`
- `batch.cancelled`

sequence 可以帮助前端去重、排序和重连；run ID 可以把反馈、日志和消息串起来。

### 8.3 会话事实源

推荐以 `app.db` 为产品事实源：

```text
conversations
turns / agent_runs
messages
action_events
```

LangGraph checkpoint 是执行投影：

- 可以按 conversation 删除。
- 可以从业务消息重建。
- 不直接作为 UI 历史接口。
- checkpoint schema 升级失败时，不应导致用户消息不可恢复。

## 9. 分阶段实施路线

### 阶段 0：建立安全和数据正确性底线

- 后端改为 `127.0.0.1`。
- 增加 Tauri session token 和 Origin allowlist。
- Provider / TTS 密钥响应脱敏。
- 上传增加大小限制。
- 清空 / 删除会话同步清理 checkpoint。
- 修复前端 TTS cache key。
- 为以上行为补集成测试。

验收标准：

- 非本机无法连接后端。
- 无 token 请求返回 401 / 403。
- API 永不返回完整密钥。
- 清空后 Agent 不再携带旧消息。

### 阶段 1：固定协议与 run 生命周期

- 增加 `agent_runs` / `turns`。
- SSE 事件增加 version、run ID 和 sequence。
- 增加前端 Abort / Stop。
- 定义结构化 ActionCommand，保留旧字符串 parser 作为短期兼容 adapter。
- 建立 Python schema 与 TypeScript 生成流程。

验收标准：

- 任一消息可追溯到 run。
- 重试不会重复消息。
- stream 中断后 run 状态明确为 failed / cancelled。

### 阶段 2：实现 VRM Director

- 单消费者队列。
- motion / say / wait / camera 完成语义。
- run 抢占和取消。
- idle / thinking / acting / speaking 状态机。
- TTS 从 Agent tool 关键路径拆出。
- 反馈按 run / tool / command 关联。

验收标准：

- 两个连续 action batch 不会并发争抢音频。
- 新消息可在限定时间内取消旧动作。
- motion 后的 say 严格按协议执行。
- 任何命令都有 completed / failed / cancelled 结果。

### 阶段 3：整理后端用例和事务边界

- route 变薄。
- Repository 不再自行 commit。
- `AgentRunService` 统一 turn 持久化。
- 简单 CRUD 使用同步 route / threadpool，远端发现显式异步化。
- 所有路由在启动时注册，Provider 实例继续惰性加载。

验收标准：

- 一次用例只有一个明确事务拥有者。
- async route 中没有未隔离的同步网络调用。
- stream 期间不长期占用不必要的 ORM transaction。

### 阶段 4：性能与资源治理

- 修复消息 COUNT 和 cursor 分页。
- 模型和 prompt 做 run-scoped 缓存。
- TTS single-flight 和磁盘淘汰。
- memory 文件限额、锁和原子写。
- VRM canonical URL、cache / dispose 所有权。
- 关闭常驻 `preserveDrawingBuffer`，增加 DPR / FPS / visibility 策略。
- 为首次 token、首次动作、TTS、VRM frame time 建立指标。

建议指标：

| 指标 | 目标方向 |
|---|---|
| 后端健康就绪时间 | 稳定且可分阶段解释 |
| 首 token 延迟 | 不被 TTS 和资源加载阻塞 |
| 首动作延迟 | action event 到达后立即可执行 |
| TTS cache hit | 按真实音色配置统计 |
| VRM 帧时间 | P95 可观察，低端设备可降级 |
| 会话列表 SQL 数据量 | 与会话数相关，不与全部消息数相关 |
| checkpoint 孤儿数 | 0 |

### 阶段 5：清理与依赖收口

- 删除兼容 DTO 和空实现。
- 统一版本来源。
- Provider SDK 拆 optional profile。
- 发布预检纳入测试、类型、lint、i18n。
- 增加前端 Director 单元测试、Playwright 主链和 Rust 启动测试。

## 10. 不建议做的事情

- 不建议改成微服务。
- 不建议仅为了双向通信把 SSE 全面换成 WebSocket。
- 不建议一次性把同步 SQLAlchemy 全部迁移为 async ORM。
- 不建议先重排所有目录再修运行时问题。
- 不建议为了“架构纯洁”给每张表都增加 Repository / Service / DTO 三层空壳。

- 不建议让 AI 直接操作 Three.js 对象；“后端发意图、前端执行”应继续保留。
