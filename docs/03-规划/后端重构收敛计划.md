# 后端重构收敛计划

本文档用于指导从 `0a9b14132f187361258b9ddf783d6c5ab8d1d40f` 重新开始的后端重构。

本轮目标不是继续沿用此前半重构分支，也不是一次性完成全系统升级，而是在干净基线上重新规划，把已经验证有价值的成果小步迁移回来。

## 1. 基线与分支

### 1.1 当前基线

本轮重构基于以下提交开始：

```text
0a9b14132f187361258b9ddf783d6c5ab8d1d40f
```

该版本作为“干净基线”，用于避免继续在前端对话页已经发生较大变化的半重构状态上修补。

### 1.2 新分支

当前重构分支：

```text
codex/backend-rebuild-from-0a9b141
```

该分支只承载后端优先的重构计划和后续小步迁移，不直接整包合入此前快照分支。

## 2. 为什么重新开始

此前重构分支已经验证了一些方向，但也引入了过大的前后端联动改动。

主要问题：

- 对话页面发生较大变化，影响当前功能完整性。
- 系统规划范围过宽，启动优化、Agent Stream、CapabilityRegistry、VRM 命令链、会话快照、资源管理等被混在同一轮推进。
- 部分抽象还没有形成稳定边界，继续修补会扩大不确定性。
- 前端主界面和后端运行时同时改动，难以判断问题来源。

因此，本轮采用：

```text
回到干净基线
 -> 只迁移已验证价值
 -> 后端优先
 -> 每一步有验收
 -> 再决定是否碰前端页面结构
```

## 3. 本轮保留的目标

本轮只保留此前验证过、且对系统稳定性有直接价值的三类成果。

### 3.1 启动速度优化

保留方向：

- 减少 `import main` 阶段的重依赖。
- 将重能力导入下沉到真正使用时。
- 区分关键路由和可延后注册的资源路由。
- 记录启动耗时快照，方便后续比较。

不保留方向：

- 不在第一阶段引入完整能力注册表。
- 不为了目录结构重组而搬动大量模块。

### 3.2 Agent Stream 主链

保留方向：

- 新增独立的 `/api/v1/agent-stream`。
- 逐步对齐 LangGraph streaming v2 事件模型。
- 保持 `conversation_id == thread_id`。
- 先让文本聊天链路稳定，再考虑 VRM 命令链。

不保留方向：

- 不删除旧聊天接口，直到新链路通过前后端验收。
- 不立即重写对话页面。
- 不把 VRM 命令链和文本流式主链绑定在同一个迁移步骤中。

### 3.3 多环境启动稳定性

保留方向：

- Tauri 动态端口。
- Tauri 向后端注入运行时目录。
- Python 后端统一读取运行时环境。
- 区分开发模式、安装版、便携版的数据和日志目录。

不保留方向：

- 不在第一轮改变用户可见启动体验。
- 不立即改成窗口完全不等待后端健康检查，除非先补足前端启动状态页。

## 4. 本轮暂停的内容

以下内容本轮明确暂停，不进入第一轮后端重构：

- 会话快照表。
- `system_prompt_snapshot`。
- `toolset_snapshot`。
- 长会话搜索、FTS、压缩。
- 资源 manifest / materialization。
- 完整 `CapabilityRegistry` 生命周期管理。
- 完整能力 adapter 目录结构。
- 完整 VRM Director / 前端命令队列。
- 对话页面大改。
- 大范围目录重组。

这些不是否定方向，而是暂时不作为当前施工范围。

- 角色提示词仍按当前角色配置生效。

## 6. 推荐实施阶段

### 阶段 A：建立重构基线

目标：

- 确认 `0a9b141` 基线可启动、可测试。
- 记录当前测试状态。
- 保留旧功能不动。

任务：

- 跑后端测试或最小 import/startup 检查。
- 跑前端 type-check / lint。
- 手动确认当前对话页面功能。

完成标准：

- 知道基线本身是否健康。
- 不引入任何功能改动。

### 阶段 B：多环境启动稳定性

目标：

- 先迁移端口、目录、运行模式相关能力。
- 不触碰对话页面。

任务：

- 迁移 Tauri 动态端口。
- 迁移运行时目录注入。
- 迁移 Python 侧运行时解析。
- 补健康检查启动快照。

完成标准：

- `uv run main.py` 可用。
- `npm run tauri:dev` 可用。
- `/api/v1/health` 可用。
- 开发/打包目录策略清晰。

### 阶段 C：后端 Agent Stream

目标：

- 新增后端 `/agent-stream`，但保留旧消息接口。

任务：

- 新增 stream schema。
- 新增 `/api/v1/agent-stream`。
- 在 `AgentCoordinator` 中增加新流式方法。
- 添加后端测试。

完成标准：

- 新旧消息接口可并存。
- 新接口可测试。
- 不改变前端主对话页面。

### 阶段 D：前端最小接入

目标：

- 用最小方式验证新 stream，不重写对话页结构。

任务：

- 新增独立 hook 或调试开关。
- 保持旧 UI 主链可退回。
- 验证文本流式输出。

完成标准：

- 新 stream 可用。
- 旧聊天 UI 不被破坏。
- 若切换失败，可以快速退回旧链路。

### 阶段 E：再评估 VRM 命令链

目标：

- 在文本 stream 稳定后，再决定是否迁移 `perform_actions`。

任务：

- 后端先注册工具。
- 前端只做命令解析和执行，不改页面布局。
- 保留旧 VRM 行为直到新链路验收。

完成标准：

- AI 命令能到前端。
- 前端能执行最小命令集。
- 失败时不影响普通文本聊天。

## 7. 每阶段必须回答的问题

每完成一个阶段，必须回答：

1. 原有功能是否仍可用？
2. 新增链路是否有独立测试或手动验收？
3. 是否引入了新旧双轨？如果有，退役条件是什么？
4. 是否改动了前端主页面？如果改了，是否必要？
5. 是否扩大了本阶段范围？

如果答案不清楚，不进入下一阶段。

## 8. 当前第一步

当前第一步不是迁移代码，而是确认基线状态：

```text
uv run pytest
npm --prefix frontend run type-check
npm --prefix frontend run lint
npm --prefix frontend run check-i18n
```
