# 开发指南

本文档用于说明 ATRI Chat 的本地开发、调试与构建流程。若只想快速找到相关文档，请先查看 [文档中心](../README.md)。

## 🛠️ 前置要求

在开始之前，请确保你的系统已安装：
- **Python 3.12+** (推荐)
- **Node.js 18+**
- **Rust** 工具链 (用于 Tauri 构建)
- **FFmpeg** (用于音频编解码处理)
- [uv](https://github.com/astral-sh/uv) (强烈推荐，用于 Python 依赖管理)

---

## 🏃 快速启动 (开发模式)

### 1. 克隆并进入目录
```bash
git clone https://github.com/1sunxiaoshou/atri-chat.git
cd atri-chat
```

### 2. 后端服务 (FastAPI)
我们使用 `uv` 管理依赖，这会比 `pip` 快得多。
```bash
# 安装依赖并同步虚拟环境
uv sync

# 启动后端 (默认端口 9099)
uv run main.py
```

### 3. 前端界面 (React + Vite)
```bash
cd frontend
npm install
npm run dev
```
访问控制台输出的地址即可开始调试。

## 📦 打包发布 (Production Build)

我们提供了一个统一的发布脚本 `scripts/release.py`，它可以一键完成前端编译、后端打包、Tauri 客户端封装及 ZIP 发布包生成。

关于安装版与便携版的数据目录策略、运行模式划分、发布文件命名和版本迭代规则，请参考：
- [桌面构建与发布指南](./桌面构建与发布指南.md)

### 统一构建工具用法
```bash
# 执行完整流水线（交互模式，会提示输入版本号）
uv run python scripts/release.py

# 命令行指定版本打包完整版
uv run python scripts/release.py release --format all --app-version 1.0.1

# 发布前预检
uv run python scripts/release.py check

# 仅清理构建产物
uv run python scripts/release.py clean

# 同步并检查发布版本
uv run python scripts/bump_version.py 1.0.2
uv run python scripts/bump_version.py --check
```

### 构建产出说明
执行完成后，根目录的 `release_package/` 文件夹下会生成：
- `atri-chat-{version}-windows-x64-setup.exe`：安装版程序。
- `atri-chat-{version}-windows-x64-portable.zip`：绿色便携版压缩包。
- `atri-chat-{version}-sha256sums.txt`：发布文件校验清单。

## 🏗️ 项目架构

- **`core/`**: 系统核心逻辑，包括 ASR, TTS, VRM 调度。
- **`api/`**: 暴露给前端的 RESTFUL 接口。
- **`frontend/`**: 基于 React 19 的单页应用。
- **`frontend/src-tauri/`**: Rust 壳程序，负责窗口管理及后端进程守护。
- **`data/`**: (运行时生成) 所有的用户数据、模型、配置均存放在此，实现真正的 Portable。

## ✅ 当前架构验证入口

在第一轮 `useStream` 主骨架改造后，建议至少按下面顺序验证启动链、后端流式链和前端联调面。

### 1. 后端自动化测试

```bash
uv run pytest tests/test_health_startup.py tests/test_agent_stream_route.py -q
```

覆盖内容：
- `/api/v1/health` 返回启动阶段快照
- `agent-stream` 返回 `metadata / messages / updates / custom`
- `agent-stream` 的运行时异常会转成 `custom error`
- 非 `text` 模式请求会被正确拒绝
- `/api/v1/runtime/status` 返回统一的能力状态快照

### 2. 真实模型端到端流式测试

推荐使用环境变量传递 API Key，避免把密钥写入代码或日志。

```bash
# PowerShell
$env:QWEN_API_KEY="你的密钥"
$env:LOG_LEVEL="WARNING"
$env:ENABLE_HTTP_LOGGING="false"
$env:ENABLE_LLM_CALL_LOGGER="false"
uv run python tests/agent_stream_e2e.py --model-id qwen3.5-flash
```

如果希望通过 `pytest` 触发同一条真实链路：

```bash
# PowerShell
$env:QWEN_API_KEY="你的密钥"
$env:LOG_LEVEL="WARNING"
$env:ENABLE_HTTP_LOGGING="false"
$env:ENABLE_LLM_CALL_LOGGER="false"
uv run pytest tests/test_agent_stream_e2e.py -m integration -q
```

覆盖内容：
- 真实模型配置可用
- `conversation_id == thread_id`
- 新 `/api/v1/agent-stream` 链路可稳定返回 `messages / updates / custom / metadata`

### 3. 前端联调与构建检查

```bash
npm --prefix frontend run type-check
npm --prefix frontend run lint
npm --prefix frontend run check-commands
npm --prefix frontend run build
```

覆盖内容：
- `useAgentStream` 类型与调用面稳定
- 命令系统第一版解析器可用
- `ChatInterface` 文本模式接线正常
- 生产构建不因新流式主链报错

### 4. 手工联调建议

- 启动后访问 `/api/v1/health`，确认返回 `startup.phases_ms` 和 `health_ready_ms`
- 访问 `/api/v1/runtime/status`，确认 `agent / asr / tts / vrm` 四类能力状态可见
- 文本模式下发送一条消息，确认前端能看到流式文本
- 带工具调用的回复，确认工具卡片状态能跟随变化
- 异常场景下确认前端收到 `custom error`，而不是直接断流
