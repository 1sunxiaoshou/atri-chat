# 日志系统设计

## 📋 设计理念

### 核心原则

1. **分离关注点** - 不同类型的日志记录到不同文件
2. **性能优先** - 异步写入，避免阻塞请求
3. **便于分析** - 结构化格式，易于解析和查询
4. **适度保留** - 根据重要性设置不同的保留期限

### 日志的目的

| 目的     | 说明                   | 对应日志             |
| -------- | ---------------------- | -------------------- |
| 问题排查 | 快速定位生产环境 bug   | error.log, app.log   |
| 性能监控 | 识别慢接口和瓶颈       | slow.log, access.log |
| 流量分析 | 统计访问量和用户行为   | access.log           |
| 安全审计 | 追踪异常访问和敏感操作 | access.log, app.log  |

## 📁 日志文件说明

### 1. access.log - 访问日志

**目的**: 记录所有 HTTP 请求，类似 Nginx 的 access.log

**格式**:
```
时间 | IP地址 | [请求ID] | 方法 路径 | 状态码 | 耗时 | User-Agent
```

**示例**:
```
2026-03-02 10:30:15 | 192.168.1.100 | [a1b2c3d4] | GET    /api/v1/characters | 200 | 15ms | Mozilla/5.0...
2026-03-02 10:30:16 | 192.168.1.100 | [e5f6g7h8] | POST   /api/v1/messages   | 200 | 1250ms | Mozilla/5.0...
```

**用途**:
- 流量统计和分析
- 用户行为追踪
- 异常访问检测
- 性能趋势分析

**保留期限**: 7 天

---

### 2. app.log - 应用日志

**目的**: 记录应用级别的所有日志（INFO 及以上）

**格式**:
```
时间 | 级别 | 模块:函数:行号 - 消息
```

**示例**:
```
2026-03-02 10:30:15.123 | INFO     | core.services.conversation_service:create_conversation:45 - 创建新会话: conv-123
2026-03-02 10:30:16.456 | WARNING  | core.middleware.logging_middleware:dispatch:78 - [e5f6g7h8] POST /api/v1/messages - 400 (50ms) | IP: 192.168.1.100
2026-03-02 10:30:17.789 | ERROR    | core.tts.factory:create_tts:120 - TTS 创建失败: 供应商未启用
```

**用途**:
- 完整的应用运行记录
- 问题排查和调试
- 业务逻辑追踪

**保留期限**: 7 天

---

### 3. error.log - 错误日志

**目的**: 单独记录所有错误，包含完整堆栈信息

**格式**:
```
时间 | 级别 | 模块:函数:行号 - 消息
堆栈信息
```

**示例**:
```
2026-03-02 10:30:20.123 | ERROR    | core.agent_coordinator:send_message:234 - [f9g0h1i2] POST /api/v1/messages - Exception (1500ms): Connection timeout | IP: 192.168.1.100
Traceback (most recent call last):
  File "core/agent_coordinator.py", line 234, in send_message
    response = await agent.ainvoke(...)
  ...
TimeoutError: Connection timeout
```

**用途**:
- 快速查看所有错误
- 错误趋势分析
- 告警和监控

**保留期限**: 30 天（错误日志保留更久）

---

### 4. slow.log - 慢查询日志

**目的**: 记录响应时间超过阈值（默认 1000ms）的请求

**格式**:
```
时间 | [请求ID] | 方法 路径 | 耗时 ⚠️
```

**示例**:
```
2026-03-02 10:30:16 | [e5f6g7h8] | POST /api/v1/messages | 1250ms ⚠️
2026-03-02 10:35:22 | [i3j4k5l6] | GET /api/v1/conversations/123/messages | 2100ms ⚠️
```

**用途**:
- 性能优化的重点目标
- 识别需要优化的接口
- 监控性能退化

**保留期限**: 30 天

**阈值配置**: 在 `core/middleware/logging_middleware.py` 中修改 `SLOW_THRESHOLD_MS`

---

## 🎯 日志级别策略

### 开发环境

**控制台输出**: DEBUG 级别，彩色显示
- 显示所有日志，方便调试
- 包含详细的模块和函数信息

**文件输出**:
- `app.log`: INFO 及以上
- `access.log`: 所有请求
- `error.log`: 错误和异常
- `slow.log`: 慢查询

### 生产环境

**控制台输出**: WARNING 级别，简洁格式
- 只显示警告和错误，减少噪音
- 避免敏感信息泄露

**文件输出**: 与开发环境相同
- 完整记录到文件，便于事后分析

## 🔧 配置说明

### 环境变量

在 `.env` 文件中配置：

```bash
# 运行环境
ENV=development  # development, production, staging

# 日志级别（可选，默认根据环境自动设置）
LOG_LEVEL=INFO   # TRACE, DEBUG, INFO, WARNING, ERROR, CRITICAL
```

### 默认日志级别

| 环境        | 默认级别 | 说明               |
| ----------- | -------- | ------------------ |
| development | DEBUG    | 详细日志，方便调试 |
| staging     | INFO     | 中等日志，测试环境 |
| production  | WARNING  | 精简日志，生产环境 |

### 慢查询阈值

在 `core/middleware/logging_middleware.py` 中修改：

```python
# 慢查询阈值（毫秒）
SLOW_THRESHOLD_MS = 1000  # 默认 1 秒
```

## 📊 日志分析示例

### 1. 查看最近的错误

```bash
tail -f logs/error.log
```

### 2. 统计接口访问量

```bash
# 统计各接口的访问次数
cat logs/access.log | awk '{print $7}' | sort | uniq -c | sort -rn

# 输出示例：
# 1250 /api/v1/characters
#  890 /api/v1/conversations
#  567 /api/v1/messages
```

### 3. 查找慢查询

```bash
# 查看所有慢查询
cat logs/slow.log

# 统计慢查询最多的接口
cat logs/slow.log | awk '{print $5}' | sort | uniq -c | sort -rn
```

### 4. 追踪特定请求

```bash
# 通过请求 ID 追踪完整流程
grep "a1b2c3d4" logs/*.log

# 输出示例：
# logs/access.log:2026-03-02 10:30:15 | 192.168.1.100 | [a1b2c3d4] | GET /api/v1/characters | 200 | 15ms
# logs/app.log:2026-03-02 10:30:15.123 | INFO | [a1b2c3d4] 查询角色列表
```

### 5. 分析错误率

```bash
# 统计各状态码的数量
cat logs/access.log | awk '{print $9}' | sort | uniq -c

# 输出示例：
# 9500 200
#  350 404
#   50 500
```

## 🚀 性能优化

### 1. 异步写入

所有文件日志使用 `enqueue=True` 参数，异步写入，避免阻塞请求：

```python
logger.add(
    LOG_DIR / "app.log",
    enqueue=True,  # 异步写入
    ...
)
```

### 2. 日志轮转

自动轮转日志文件，避免单个文件过大：

```python
logger.add(
    LOG_DIR / "app.log",
    rotation="100 MB",  # 文件达到 100MB 时轮转
    ...
)
```

### 3. 自动清理

自动删除过期日志，节省磁盘空间：

```python
logger.add(
    LOG_DIR / "app.log",
    retention="7 days",  # 只保留 7 天
    ...
)
```

## 📈 监控和告警

### 推荐工具

1. **日志聚合**: ELK Stack (Elasticsearch + Logstash + Kibana)
2. **日志分析**: Grafana Loki
3. **告警**: Prometheus + Alertmanager
4. **简单方案**: 定时脚本 + 邮件/钉钉通知

### 告警规则示例

```bash
#!/bin/bash
# 检查错误日志，如果最近 5 分钟有超过 10 个错误，发送告警

ERROR_COUNT=$(tail -n 1000 logs/error.log | grep "$(date -d '5 minutes ago' '+%Y-%m-%d %H:%M')" | wc -l)

if [ $ERROR_COUNT -gt 10 ]; then
    echo "⚠️ 错误数量异常: $ERROR_COUNT 个错误" | mail -s "日志告警" admin@example.com
fi
```

## 🔒 安全和隐私

### 敏感信息脱敏

在记录日志时，自动脱敏敏感信息：

```python
# 示例：脱敏 API Key
def mask_sensitive(text: str) -> str:
    """脱敏敏感信息"""
    # API Key: sk-xxx... → sk-***
    text = re.sub(r'(sk-)[a-zA-Z0-9]{10,}', r'\1***', text)
    # 密码: password=xxx → password=***
    text = re.sub(r'(password=)[^\s&]+', r'\1***', text)
    return text

logger.info(mask_sensitive(f"API 调用: {api_key}"))
```

### 访问控制

生产环境日志文件权限设置：

```bash
# 只允许应用用户读写
chmod 600 logs/*.log
chown app:app logs/*.log
```

## 📚 最佳实践

### 1. 日志内容

✅ **应该记录**:
- 请求 ID（用于追踪）
- 关键业务操作（创建、更新、删除）
- 错误和异常（包含堆栈）
- 性能指标（响应时间）
- 安全事件（登录、权限变更）

❌ **不应该记录**:
- 用户密码
- API 密钥（完整）
- 个人敏感信息（身份证、手机号）
- 大量重复的调试信息

### 2. 日志级别使用

| 级别     | 使用场景       | 示例                     |
| -------- | -------------- | ------------------------ |
| DEBUG    | 详细的调试信息 | 变量值、函数调用         |
| INFO     | 关键业务操作   | 用户登录、创建订单       |
| WARNING  | 可恢复的异常   | 重试成功、降级处理       |
| ERROR    | 需要关注的错误 | API 调用失败、数据库错误 |
| CRITICAL | 系统级别故障   | 服务崩溃、数据丢失       |

### 3. 结构化日志

使用 `extra` 参数添加结构化信息：

```python
logger.info(
    "用户登录成功",
    extra={
        "user_id": user.id,
        "ip": request.client.host,
        "user_agent": request.headers.get("User-Agent"),
    }
)
```

## 🎓 总结

### 成熟项目的日志系统特点

1. **分层记录** - 不同类型的日志分开存储
2. **结构化** - 易于解析和查询
3. **高性能** - 异步写入，不阻塞业务
4. **可追踪** - 请求 ID 贯穿整个调用链
5. **可监控** - 支持告警和可视化
6. **安全** - 敏感信息脱敏
