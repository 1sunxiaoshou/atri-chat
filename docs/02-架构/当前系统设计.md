# 当前系统设计

本文档记录当前代码实际落地的系统设计，用于帮助开发者快速理解 ATRI Chat 的运行链路、模块边界和当前重构状态。

本文不是未来蓝图。未来演进仍以 `docs/03-规划/` 下的规划文档为参考；如果规划与本文冲突，说明规划尚未落地，当前开发应优先以本文和代码为准。

## 1. 系统定位

ATRI Chat 当前是一个 Windows 桌面优先的 AI 伴侣应用，核心能力包括：

- 角色化 LLM 聊天
- 长期记忆工具
- VRM 3D 角色渲染与动作控制
- ASR 语音识别
- TTS 语音合成
- 本地 SQLite 状态与资源管理

整体形态可以理解为：

```text
Tauri 桌面壳
 -> React 前端
 -> FastAPI 控制面
 -> LangChain / LangGraph Agent
 -> SQLite / 文件资源 / ASR / TTS / VRM 前端执行层
```

## 2. 当前分层

当前系统已经形成了较清晰的五层结构，但并非所有层都已经完全抽象化。

| 层 | 当前职责 | 主要位置 |
|---|---|---|
| 桌面壳 | 启动后端进程、分配端口、注入运行时目录 | `frontend/src-tauri/` |
| 前端界面 | 聊天、角色管理、模型设置、VRM 渲染、状态展示 | `frontend/` |
| 控制面 API | 健康检查、路由注册、资源/配置/会话接口 | `main.py`、`api/routes/` |
| Agent 运行时 | LLM 调用、工具过滤、提示词构建、消息持久化 | `core/agent_coordinator.py` |
| 状态与资源 | SQLite 表、长期记忆文件、静态资源目录、运行状态 | `core/db/`、`data/`、`core/runtime_status.py` |

当前需要注意的是：`CapabilityRegistry` 已经有最小落地版本，但还主要承担“状态聚合、预热和反馈记录”的职责，并没有完全替代各能力原有的服务/工厂/前端执行逻辑。

## 3. 启动链路

### 3.1 桌面端启动

Tauri 的入口位于 `frontend/src-tauri/src/lib.rs`。

当前流程：

1. 解析运行时布局。
2. 创建数据目录和日志目录。
3. 分配后端端口。
4. 启动后端：
   - 开发模式：`uv run python main.py`
   - 打包模式：启动 sidecar 后端二进制
5. 等待 `/api/v1/health` 返回 200。
6. 向前端暴露 `get_backend_port`，前端据此构造 API 地址。

运行时目录策略由前后端共同维护：

- Rust 侧：`frontend/src-tauri/src/runtime.rs`
- Python 侧：`core/runtime.py`、`core/config.py`

Tauri 会向后端注入这些环境变量：

- `ATRI_BACKEND_PORT`
- `ATRI_RUNTIME_MODE`
- `ATRI_APP_ENV`
- `ATRI_APP_ROOT`
- `ATRI_DATA_ROOT`
- `ATRI_LOGS_ROOT`

### 3.2 后端启动

后端入口是 `main.py`。

当前启动重点：

- `.env` 只加载一次，且不覆盖 Tauri 注入的运行时端口。
- `settings.ensure_directories()` 在模块加载早期执行，保证静态资源挂载目录存在。
- FastAPI lifespan 中并行执行：
  - 注册关键路由
  - 初始化业务数据库
  - 初始化 LangGraph checkpointer
- 非关键资源路由在后台延迟注册。
- 启动阶段通过 `core/startup_metrics.py` 记录耗时快照。
- 启动后调度能力后台预热。

关键路由当前包括：

- `/api/v1/health`
- `/api/v1/runtime/status`
- `/api/v1/runtime/events`
- `/api/v1/agent-stream`
- `/api/v1/characters`
- `/api/v1/conversations`
- `/api/v1/models`
- `/api/v1/providers`
- `/api/v1/asr`
- `/api/v1/tts`

延迟注册的路由主要是资源管理类接口，例如 avatar、motion、voice、upload 等。

## 4. 数据与目录

### 4.1 数据目录

当前数据目录由 `core/config.py` 统一派生。

重要目录：

| 目录 | 用途 |
|---|---|
| `data/sqlite/` | 应用数据库、LangGraph checkpoints、长期 store |
| `data/memory/` | 角色长期记忆 Markdown 文件 |
| `data/models/asr/` | ASR 模型资源 |
| `data/assets/images/` | 图片资源 |
| `data/assets/vrm/models/` | VRM 模型 |
| `data/assets/vrm/motions/` | VRM 动作 |
| `data/assets/vrm/thumbnails/` | VRM 缩略图 |
| `data/logs/` | 日志 |

### 4.2 主要数据库对象

主要 ORM 定义在 `core/db/models.py`。

当前核心表：

- `provider_configs`：LLM 供应商配置
- `models`：模型配置和能力标记
- `characters`：角色配置，组合提示词、模型、形象和音色
- `conversations`：会话
- `messages`：消息
- `assets_avatars`：VRM 形象资产
- `assets_motions`：动作资产
- `character_motion_bindings`：角色动作绑定
- `tts_providers`：TTS 供应商
- `assets_voices_v2`：音色资产

当前尚未落地为正式表的规划对象包括：

- `conversation_sessions`
- `system_prompt_snapshot`
- `toolset_snapshot`
- `agent_run_events`
- `asset_manifests`
- `asset_materializations`

这些仍属于后续规划，不应在当前代码理解中当作已完成能力。

## 5. 聊天主链

当前聊天主链已经从旧 `/messages` SSE 迁移到：

```text
前端 useAgentStream
 -> POST /api/v1/agent-stream
 -> AgentCoordinator.stream_runtime_events()
 -> LangChain / LangGraph Agent
 -> SSE event
 -> 前端 useStream
```

### 5.1 前端入口

主要文件：

- `frontend/components/chat/ChatInterface.tsx`
- `frontend/hooks/useAgentStream.ts`
- `frontend/services/api/messages.ts`

`ChatInterface` 负责：

- 当前会话、角色、模型参数的组合
- 普通聊天和 VRM 聊天模式切换
- 发送消息
- 将 VRM custom event 转交给 VRM 执行层
- 普通模式下触发自动 TTS 播放

`useAgentStream` 负责：

- 通过 `@langchain/react` 的 `useStream` 连接 `/agent-stream`
- 把 LangChain 消息转换为前端 `Message`
- 维护流式 reasoning、tool call、custom event
- 加载历史消息
- 处理标题更新、VRM 命令、镜头命令和错误事件

### 5.2 后端入口

主要文件：

- `api/routes/agent_stream.py`
- `api/schemas.py`
- `core/agent_coordinator.py`

`/api/v1/agent-stream` 接收 `AgentStreamRequest`：

- `input.messages`：来自前端的 LangGraph 风格消息输入
- `context`：ATRI 的运行时上下文，如会话、角色、模型、显示模式
- `config.configurable.thread_id`：LangGraph thread id

当前约定：

```text
conversation_id == thread_id
```

这使业务会话、LangGraph checkpoint 和前端 thread 使用同一个标识。

### 5.3 AgentCoordinator

`core/agent_coordinator.py` 是当前 Agent 主协调器。

当前职责：

- 惰性创建 LangChain `create_agent`
- 接入 checkpointer 和 store
- 组装工具：
  - 长期记忆工具
  - VRM 命令工具
- 通过 middleware 动态选择模型、提示词和工具
- 保存用户消息和助手消息
- 将 LangGraph stream part 转换为前端可消费的 SSE payload

当前 Agent 实例是单例，但不同会话依靠 `thread_id` 隔离。

## 6. 提示词与工具

### 6.1 提示词

当前提示词服务位于：

- `core/prompts/service.py`
- `core/prompts/templates/identity.md`
- `core/prompts/templates/normal.md`
- `core/prompts/templates/vrm.md`

提示词由 `PromptService.build_system_prompt()` 根据角色和模式构建。

当前模式：

- 文本模式：角色身份 + 普通聊天模式提示词
- VRM 模式：角色身份 + VRM 命令模式提示词

虽然规划中提到“会话开始时冻结 system prompt snapshot”，但当前代码仍是在运行时通过 dynamic prompt middleware 构建提示词，尚未落地会话快照表。

### 6.2 动态模型

当前模型选择通过 `core/middleware/dynamic_model.py` 完成。

Agent 创建时使用占位模型；每次请求通过运行时上下文中的：

- `provider_config_id`
- `model_id`
- `model_kwargs`

动态创建真实模型实例。

### 6.3 工具过滤

当前工具过滤在 `core/middleware/dynamic_tools.py`。

规则：

- 所有模式允许 `memory_*`
- 仅 VRM 模式允许：
  - `perform_actions`
  - `control_camera`
  - 未来 `vrm_*` 工具

这样可以避免普通文本模式暴露过宽工具面。

### 6.4 长期记忆工具

长期记忆工具位于 `core/tools/memory_tools_v3.py`。

当前提供：

- `memory_read`
- `memory_write`
- `memory_search`
- `memory_list`

记忆以角色为单位存放在：

```text
data/memory/{character_id}/
```

工具只允许访问角色自己的相对路径，默认使用 Markdown 文件。

## 7. VRM 命令链

当前 VRM 控制不是后端直接操作渲染对象，而是：

```text
AI 调用工具
 -> 后端提取工具调用
 -> custom event 下发前端
 -> 前端解析命令
 -> useVRM 执行表情、字幕、动作、镜头
 -> 前端回报执行结果
 -> runtime status 记录反馈
```

### 7.1 后端工具

工具位于 `core/tools/action_tools.py`。

当前工具：

- `perform_actions(commands: string[])`
- `control_camera(action, preset, durationMs?)`

`perform_actions` 当前允许的命令：

```text
emotion <name>
motion <name>
say <emotion> | <text>
wait <ms>
```

后端工具只做基础校验，并返回 `ok` 或错误字符串。真正执行发生在前端。

### 7.2 VRM 输出策略

VRM 模式使用 `core/services/output_strategy.py` 中的 `VRMOutputStrategy`。

当前 VRM 模式不是 token 级文本流式输出，而是：

1. 使用 `agent.ainvoke()` 获得完整结果。
2. 保存助手自然语言回复。
3. 从 tool calls 中提取：
   - `vrm_commands`
   - `camera_command`
4. 通过 custom event 发给前端。

### 7.3 前端执行

主要文件：

- `frontend/utils/actionCommands.ts`
- `frontend/components/vrm/logic/streamAdapter.ts`
- `frontend/components/vrm/logic/useVRM.ts`
- `frontend/store/vrm/useVRMStore.ts`

前端执行规则：

- `emotion`：立即切换表情
- `motion`：按动作 ID 或名称在角色 reply 动作中查找并播放
- `say`：设置表情和字幕，并按文本长度等待一段时间
- `wait`：插入停顿
- `control_camera`：写入相机控制状态，由 R3F 侧消费

执行完成后，前端调用：

```text
POST /api/v1/runtime/vrm-feedback
```

将执行结果和当前状态回传后端。

## 8. 运行时状态

当前运行时状态位于：

- `core/runtime_status.py`
- `api/routes/runtime.py`
- `frontend/hooks/useRuntimeStatus.ts`
- `frontend/components/layout/RightSidebar.tsx`

### 8.1 CapabilityRegistry 当前职责

当前 `CapabilityRegistry` 负责：

- 注册能力 manifest
- 聚合能力状态
- 调度后台预热
- 记录前端 VRM 执行反馈
- 保存最近的能力事件到内存队列

当前注册的能力：

- `agent`
- `asr`
- `tts`
- `vrm`

状态取值包括：

- `disabled`
- `uninitialized`
- `warming`
- `ready`
- `busy`
- `failed`

### 8.2 状态接口

当前接口：

- `GET /api/v1/runtime/status`
- `GET /api/v1/runtime/events`
- `POST /api/v1/runtime/vrm-feedback`

前端右侧设置面板会轮询 runtime status，展示控制面状态、能力状态、health ready 时间和 ready 能力数量。

当前注意点：

- 状态事件保存在内存中，进程重启后丢失。
- `CapabilityRegistry` 尚未持久化 `agent_run_events`。
- 能力生命周期接口还不完整，部分能力仍通过原有工厂或前端 hook 管理。

## 9. ASR 与 TTS

### 9.1 ASR

ASR 相关位置：

- `api/routes/asr.py`
- `api/routes/asr_mgmt.py`
- `core/asr/`
- `core/dependencies.py`
- `core/runtime_status.py`

当前 ASR 引擎通过 `get_asr_engine()` 惰性获取。

运行时状态会检查：

- SenseVoice 模型是否安装
- int8/fp32 资源是否存在
- ASR 引擎是否已经缓存

ASR 管理接口负责模型下载、状态查询和清理。

### 9.2 TTS

TTS 相关位置：

- `api/routes/tts.py`
- `api/routes/tts_providers.py`
- `api/routes/voice_assets.py`
- `core/tts/`
- `frontend/hooks/useTTS.ts`

当前 TTS 采用供应商配置 + 音色资产的结构：

- `tts_providers` 保存供应商级配置
- `assets_voices_v2` 保存音色级配置
- 角色通过 `voice_asset_id` 引用音色

`TTSFactory` 负责创建和缓存 TTS 实例。runtime status 会根据供应商数量和实例缓存判断 TTS 状态。

## 10. 前端应用结构

前端入口：

- `frontend/index.tsx`
- `frontend/App.tsx`

当前前端启动先执行 `initApiConfig()`：

- Tauri 环境：调用 `get_backend_port`
- Web 开发环境：使用 `VITE_API_BASE_URL` 或默认 `9099`
- 生产 Web：使用同源地址

`App.tsx` 负责顶层布局和视图切换：

- `chat`
- `admin`
- `characters`
- `settings`

主要状态：

- 顶层会话列表在 `App.tsx`
- 角色、模型、供应商数据在 `useDataStore`
- 聊天流状态在 `useAgentStream`
- VRM 渲染状态在 `useVRMStore`
- 音频播放状态在 `useAudioStore`

## 11. 当前已落地与未落地

### 11.1 已落地

- 后端关键路由与延迟路由拆分
- 启动耗时快照
- `/health` 返回启动快照
- `agent_stream + useAgentStream` 主链
- `conversation_id == thread_id`
- 文本模式 LangGraph v2 stream 消费
- VRM `perform_actions / control_camera` 最小命令链
- 前端 VRM 执行反馈
- 最小 `CapabilityRegistry`
- 右侧状态面板展示能力状态
- PromptService 模板化提示词
- ASR/TTS 惰性实例和状态聚合

### 11.2 部分落地

- Capability 生命周期管理：已有状态和预热，但还没有完全统一所有能力的创建、关闭和重置。
- 运行时事件：已有内存事件队列，但尚未持久化。
- VRM Director：已有前端执行链，但还不是独立完整的 Director / 队列系统。
- Prompt 稳定化：已有简化模板，但尚未形成会话快照。

### 11.3 尚未落地

- `conversation_sessions`
- `system_prompt_snapshot`
- `toolset_snapshot`
- `capability_profile_snapshot`
- 长会话搜索、FTS 和压缩
- 资源 manifest / materialization
- 持久化 `agent_run_events`
- 完整能力适配器目录结构

## 12. 当前开发判断准则

在后续整理中，建议按以下规则判断保留、暂停或回退：

| 类型 | 建议 |
|---|---|
| 已经成为主链且可运行 | 保留 |
| 能运行但命名或目录不理想 | 先保留，记录后续整理 |
| 新旧双轨同时存在 | 选择一条主链，另一条列为清理项 |
| 只有规划但代码未闭环 | 暂停推进 |
| 半成品抽象且未被主链使用 | 回退或隔离 |
| 大范围目录搬迁 | 暂缓，等主链稳定后再做 |

当前最应优先保护的主链是：

```text
启动
 -> /health
 -> /agent-stream
 -> useAgentStream
 -> 文本聊天 / VRM 命令执行
 -> runtime status / feedback
```
